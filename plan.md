# Implementation Plan: SABO MVP Web Platform

**Branch**: `002-mvp-web-platform` | **Date**: 2026-05-19 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-mvp-web-platform/spec.md`

## Summary

SABO is a web-based consumer activism platform where users join groups organized around specific corporate grievances and coordinate financial and reputational pressure campaigns. The MVP ships when the complete core loop (Grievance → Group Formation → Tactic Selection → Action → Report Back → Social Reward → Escalation) is completable end-to-end for all four seeded campaigns (Amazon, Meta, Cigna, Wells Fargo). Key technical challenges: real-time feed updates (SSE + Redis pub/sub), constellation rollup metrics (parent/subsidiary without double-counting), and a pluggable corporate ownership integration layer (SEC EDGAR now, OpenCorporates later).

## Technical Context

**Language/Version**: Python 3.11+ (backend) | Node.js 20 LTS (frontend runtime)

**Primary Dependencies**:
- Backend: FastAPI, SQLAlchemy (async), Alembic, sse-starlette, Celery, Redis, python-jose, passlib[bcrypt], boto3, httpx (EDGAR API calls)
- Frontend: SvelteKit (adapter-node), TypeScript

**Storage**: PostgreSQL 16 (primary data), Redis 7 (pub/sub + Celery broker), Hetzner Object Storage (proof artifacts)

**Testing**: pytest + pytest-asyncio (backend), Vitest + Playwright (frontend E2E)

**Target Platform**: Linux (Hetzner VPS), Nginx reverse proxy, Cloudflare CDN

**Project Type**: Web application (REST API backend + SSR frontend)

**Performance Goals**:
- Group page load: < 3 seconds on 4G mobile (SC-005)
- Real-time update latency: ≤ 60 seconds (FR-040, SC-004)
- Feed SSE: support ≥ 1,000 concurrent connections per server instance

**Constraints**:
- No AWS, Google Cloud, Meta infrastructure at any layer (Constitution Principle X)
- All schema changes via Alembic migrations — no manual SQL on production (FR-039)
- OpenCorporates integration deferred until constitution amendment; SEC EDGAR only for MVP
- Data layer, API layer, and UI layer strictly separated (FR-037)

**Scale/Scope**: MVP targets ~10K users at launch; architecture scales horizontally via Redis pub/sub fan-out

## Key Technical Decisions (from research.md)

| Decision | Choice | Rationale |
|---|---|---|
| Backend framework | FastAPI (Python 3.11) | Async, SSE via sse-starlette, consistent with existing codebase |
| Frontend framework | SvelteKit + adapter-node | Self-hostable, mobile-first, SSR built-in, no Vercel dependency |
| Database | PostgreSQL 16 + SQLAlchemy async + Alembic | Relational model fits; migrations required by constitution |
| Real-time | SSE (sse-starlette) + Redis Pub/Sub | Simpler than WebSockets for one-directional updates; scales horizontally |
| Background jobs | Celery + Redis | EDGAR lookups are async; same Redis instance as pub/sub |
| File storage | Hetzner Object Storage via boto3 | S3-compatible, approved vendor, Cloudflare CDN |
| Auth | JWT (httpOnly cookie) + bcrypt | Stateless; no third-party OAuth (vendor constraint) |
| EDGAR integration | Direct EDGAR API + custom Exhibit 21 parser | Free, no vendor approval; pluggable adapter pattern for future OpenCorporates |
| Scoring | Integer point weights by tactic impact category | Simple, auditable, no floating point drift |
| Constellation rollup | Separate labeled components on parent group page | Never merge subsidiary actions into parent's own total |

## Implementation Phases

### Phase 0: Foundation (blocks everything)

1. Docker Compose: postgres, redis services for local dev
2. Backend project scaffold: FastAPI app factory, config (pydantic-settings), database engine (SQLAlchemy async), Redis connection pool
3. Alembic setup + first migration: all entities from data-model.md
4. Frontend project scaffold: SvelteKit, adapter-node, TypeScript, Vite, shared API types

### Phase 1: Core Data + Auth (US1 prerequisite)

1. All SQLAlchemy models (Company, Group, Grievance, Tactic, TacticGroupLink, Action, ProofArtifact, User, GroupMembership, UserGroupScore, Milestone, AcknowledgmentRecord)
2. Auth endpoints: register, login, logout, refresh (JWT + bcrypt)
3. Seed data script: 4 seeded campaigns + 10–15 tactics across ≥3 categories
4. Frontend: auth pages (register/login), auth store

### Phase 2: Core Loop (US1 — P1 gate)

1. Group API: GET /groups, GET /groups/{slug}
2. Group join: POST /groups/{slug}/join
3. Tactic API: GET /tactics, GET /tactics/{id}, GET /groups/{slug}/tactics (with routing_level separation)
4. Action submission: POST /actions (with proof upload to Hetzner Object Storage)
5. Scoring service: compute and persist UserGroupScore on action submission
6. Frontend: group detail page (grievance, member count, tactic list, acknowledgment tracker), tactic detail modal, action submission form with proof upload

### Phase 3: Real-Time Feed (US3)

1. Action feed API: GET /groups/{slug}/feed (paginated)
2. SSE endpoint: GET /groups/{slug}/feed/stream
3. Redis pub/sub: publish on action commit; fan-out to all SSE subscribers for group
4. Frontend: ActionFeed component (SSE-connected, auto-updating)
5. Score update events pushed via SSE on UserGroupScore change

### Phase 4: Leaderboard + Scoring Display (US4)

1. Leaderboard API: GET /groups/{slug}/leaderboard
2. Membership preferences: PATCH /groups/{slug}/membership (leaderboard_opt_in, public_attribution_opt_in)
3. Profile page: GET /users/me + score breakdown by group and category
4. Frontend: Leaderboard component, profile page, score breakdown widget

### Phase 5: Grievance Validation + Proposals (US5)

1. GroupProposal + Vote models (already in migration)
2. Proposal API: POST /proposals, GET /proposals, POST /proposals/{id}/upvote
3. Auto-activation logic: when upvote_count >= upvote_threshold, create Group and notify proposer
4. Duplicate detection: company name match against existing active groups
5. Frontend: proposal submission form, pending proposals list, upvote UI

### Phase 6: Constellation + EDGAR (US5, US6)

1. Constellation + ConstellationMembership models
2. EDGAR service: fetch Exhibit 21 filing, parse subsidiary list, cache as OwnershipRecord (30-day TTL)
3. Celery task: async EDGAR lookup on group proposal submission
4. Constellation offer: surface parent group suggestion in POST /proposals response
5. Constellation API: GET /groups/{slug} extended with constellation data, GET /groups/{slug} parent rollup scores
6. ConstellationPanel frontend component: parent's labeled rollup display, subsidiary link on subsidiary group pages, parent-level tactic section

### Phase 7: Conglomerate Compass (US6)

1. CompassService with OwnershipDataSource interface (adapter pattern for future OpenCorporates)
2. Compass API: GET /compass/search, GET /compass/companies/{cik}
3. SABO group link injection in compass results
4. Frontend: CompassSearch component, Compass page

### Phase 8: Acknowledgment Tracker + Win State (US7)

1. AcknowledgmentRecord model (already in migration)
2. PATCH /groups/{slug}/acknowledgment (staff only)
3. SSE `acknowledgment_update` event on tracker change
4. Milestone crossed events via SSE
5. Frontend: AcknowledgmentTracker component, milestone celebration banner

### Phase 9: Venting Feed

1. VentingPost model + migration
2. POST + GET /groups/{slug}/venting endpoints
3. Strict separation: venting posts never appear in action feed API or SSE action stream
4. Frontend: separate venting surface on group page (below action board fold)

### Phase 10: Polish + Pre-Launch

1. Nginx config: reverse proxy, SSE buffering disabled, static asset caching headers
2. Plausible analytics: snippet in SvelteKit layout (no Google Analytics)
3. Staff admin endpoints: flag review, acknowledgment update, ownership force-refresh
4. Rate limiting: 429 on auth endpoints (5 failed/15 min/IP)
5. Error handling: consistent error response format across all endpoints
6. Full integration test suite (10 quickstart scenarios)
7. Partisan-language audit of all 4 seeded grievances (manual, pre-launch gate)
8. Mobile usability pass: all pages tested at 375px viewport
9. Core loop end-to-end test for all 4 seeded campaigns (SC-009 gate)
